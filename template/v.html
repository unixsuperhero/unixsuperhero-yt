Payroll System


                class Employee
                  double calculatePay()
                  boolean storeCheck()
  1. Calculating Employee Wages for the Pay Period
  2. Issuing the Check
    • These are separate operations
    • Issuing a check doesn't need to be coupled directly to wage calculation
    • Regardless of how it's calculated, to issue a check it only needs to know the check amount
    • What if you need to issue a check for any reason other than wage, like reimbursing an expense?
    • In that case, the code to issue a check wouldn't be reusable without changes that could affect the integrity of the current system

Review

From 3 simple classes, we were able to identify 6 responsibilities:

🎳 Game
  1. Tracking the Game's Progress
  2. Calculating the Score
☎ Modem
  1. Handling the Connection
  2. Transmitting Data
💵 Payroll
  1. Calculating Wages
  2. Issuing the Check

These behaviors are all very different. Even if they are associated with the same data type.

IRL

There are many shining examples of the SRP...

UNIX/Linux

  • in a future video, i will cover the unix philosophy in depth, but one variation states:
    Write programs that do one thing and do it well
  • no single command handling all fs operations:
    • ls - list files in a directory
    • cp - copy files
    • mv - move files
    • they are simple operations, and yet they were separated...
  • grep - search/filter text
  • cron - a daemon for scheduling tasks
  • crontab - a separate command for configuring the daemon

git

  • git is made up of 135 subcommands that work together under-the-hood
  • from the git book:
    [git] has a number of subcommands that do low-level work and were designed to be chained together UNIX-style or called from scripts. These commands are generally referred to as Git’s “plumbing” commands, while the more user-friendly commands are called “porcelain” commands.
  • even if you're only familiar with the porcelain commands, those commands are just leveraging other commands that have isolated responsibilities

AWS

Tons of small tools that can work in concert to build apps f all shapes and sizes

  • like 12 database solutions
  • SQS - fifo queue
  • SNS - pub/sub system
  • ElastiCache - caching

AWS is not a single application that's trying to do everything

IRL

Regardless of your personal opinion, all of these systems are focused, robust, and composable. and they're all key tools that power modern development

Homework

the next time you're looking at some code, take a minute to list out the responsibilities of the module, class, or function you're working with.

if you find more than one, think about how you can separate them
